# 코드 수정 워크플로우 규칙

## 코드 수정 전 필수 프로세스

코드를 수정하기 전에 반드시 다음 단계를 따라야 합니다:

1. **원인 분석**: 문제의 근본 원인을 파악하고 설명합니다
2. **해결 방안 제안**: 가능한 해결 방법을 제시하고 장단점을 설명합니다
3. **사용자 승인 대기**: 사용자가 제안한 방법을 승인할 때까지 기다립니다
4. **코드 수정 실행**: 승인 후에만 실제 코드를 수정합니다

## 금지 사항

- 사용자와 상의 없이 즉시 코드를 수정하는 것
- 원인 분석 없이 바로 해결책을 적용하는 것
- 여러 해결 방법이 있을 때 사용자 선택 없이 임의로 선택하는 것

## 예외 상황

다음 경우에만 사용자 승인 없이 즉시 수정 가능합니다:
- 명백한 오타 수정 (typo)
- 사용자가 명시적으로 "수정해줘", "고쳐줘" 등의 직접적인 수정 요청을 한 경우
- 이미 합의된 spec 문서의 tasks를 실행하는 경우

## 커뮤니케이션 스타일

- 원인 분석 시: "문제를 분석해보니..." 형식으로 설명
- 해결 방안 제시 시: "다음과 같은 방법들이 있습니다..." 형식으로 옵션 제시
- 승인 요청 시: "어떤 방법으로 진행할까요?" 또는 "이렇게 수정해도 될까요?" 형식으로 질문
